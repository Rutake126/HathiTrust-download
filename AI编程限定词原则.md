# AI 编程限定词原则
> 降低 AI 编程幻觉的系统性提示词规范
> 参考 Andrej Karpathy 的提示词哲学整理

---

## 核心理念

AI 编程幻觉的根源在于：**模型在不确定时仍会自信地生成内容**。限定词原则的本质是通过结构化约束，将 AI 的"猜测空间"压缩到最小，让它只在有把握的范围内工作。

> *"Don't ask AI to be creative where you need it to be correct."*
> —— 借鉴 Karpathy 的核心观点

---

## 一、上下文锚定原则（Context Anchoring）

**原则：给 AI 提供足够的事实锚点，而非让它自行推断。**

### ✅ 正确做法

```
我有一个 Python FastAPI 项目，使用 SQLAlchemy 2.0 + PostgreSQL 15。
目前的 User 模型如下：

[粘贴实际代码]

请在此基础上添加邮箱唯一性校验。
```

### ❌ 错误做法

```
帮我给用户模型加邮箱校验。
```

### 限定词清单

| 限定词类型 | 示例表达 |
|-----------|---------|
| 语言/框架版本 | `Python 3.11`、`React 18`、`Next.js 14 App Router` |
| 已有代码 | 直接粘贴相关函数/类/文件 |
| 依赖约束 | `只使用已有依赖，不引入新库` |
| 文件路径 | `修改 src/utils/auth.ts，不要新建文件` |
| 数据库/API | `数据库是 MySQL 8，不是 PostgreSQL` |

## 二、范围收窄原则（Scope Narrowing）

**原则：任务越具体，幻觉越少。拆分大任务，逐步执行。**

### 任务拆分策略

```
❌ 一次性请求（高幻觉风险）：
"帮我写一个完整的用户认证系统"

✅ 分步请求（低幻觉风险）：
第1步："只写 JWT token 生成函数，入参是 user_id 和 expire_minutes"
第2步："只写 token 验证中间件，复用上面的函数"
第3步："只写登录路由，调用上面两个函数"
```

### 限定词模板

```
只修改 [具体函数名] 这一个函数。
不要重构其他代码。
不要添加我没有要求的功能。
改动范围限制在 [N] 行以内。
```

## 三、输出格式约束原则（Output Constraining）

**原则：明确告诉 AI 输出什么、不输出什么。**

### 常用限定词

```
只输出代码，不要解释。
只输出需要修改的部分，用注释标注修改位置。
不要省略任何代码，即使是未修改的部分也要保留。
输出完整文件，不要用 "// ... 其余代码不变" 代替。
```

### 差异输出模式（推荐用于大文件）

```
以 diff 格式输出修改内容：
- 用 "// [CHANGE START]" 和 "// [CHANGE END]" 包裹修改部分
- 保留修改前后各 3 行上下文
- 标注每处修改的原因
```

## 四、不确定性显式化原则（Uncertainty Surfacing）

**原则：强制 AI 在不确定时说出来，而非自行填充。**

### 必加的元限定词

```
如果你不确定某个 API 的具体用法，请明确说"我不确定，建议查阅文档"，不要猜测。

如果有多种实现方式，列出选项让我选择，不要自行决定。

如果我的描述有歧义，先问清楚再写代码。
```

### 针对版本敏感 API 的限定词

```
该函数是否在 [框架名] [版本号] 中存在？如果不确定，请说明。
请基于 [官方文档链接] 的内容回答，不要依赖你的训练数据。
```

## 五、验证驱动原则（Verification-First）

**原则：先定义"正确"的标准，再让 AI 生成代码。**

### 测试优先模式（参考 Karpathy TDD 思路）

```
第1步：先帮我写测试用例（不写实现）：
- 输入：[具体示例]
- 期望输出：[具体期望]
- 边界情况：[列举]

第2步：确认测试用例后，再写能通过这些测试的实现代码。
```

### 自检限定词

```
写完后，请检查：
1. 是否处理了 null/undefined 的情况？
2. 是否有可能的数组越界？
3. 异步操作是否都有错误处理？
列出你发现的潜在问题。
```

## 六、角色与知识边界原则（Role Bounding）

**原则：明确告诉 AI 它扮演什么角色，边界在哪里。**

### 角色定义模板

```
你是一个严谨的 [语言] 代码审查员。
你的任务是：[具体任务]
你的限制是：
- 只评论代码质量，不重写代码
- 不引入我未使用的设计模式
- 发现问题时给出具体行号
```

### 禁止创意的场景

```
这是生产代码，不需要"更优雅"的重构。
只修复我指出的 bug，保持其余代码风格不变。
不要"顺手"优化其他地方。
```

## 七、渐进确认原则（Incremental Confirmation）

**原则：在复杂任务中，让 AI 先描述方案，人工确认后再执行。**

### 两阶段工作流

```
阶段一（方案确认）：
"请描述你打算如何实现 [功能]，包括：
 - 涉及哪些文件
 - 新增/修改哪些函数
 - 使用什么数据结构
 不要写代码，只描述方案。"

[人工审核方案]

阶段二（执行）：
"按照上述方案执行，严格按照描述来，不要临时改变方案。"
```

## 八、幻觉高发区专项原则

### 8.1 第三方库 API

```
[库名] 的 [函数名] 方法的参数，请以官方文档为准。
不确定时，优先给出文档链接，而不是直接写代码。
```

### 8.2 时间敏感内容

```
你的训练数据截止到某个时间点。
如果这个问题涉及 [库/框架] 近期的 breaking change，请提示我验证。
```

### 8.3 复杂业务逻辑

```
这段逻辑涉及 [业务规则]。
如果逻辑推导过程中有任何假设，请显式列出假设，不要隐含。
```

### 8.4 数据库 / SQL

```
表结构如下：[粘贴 schema]
数据库版本：[MySQL/PG/SQLite] [版本]
不要使用该版本不支持的语法。
查询结果中不确定的字段含义，请标注 [需确认]。
```

## 九、实战提示词模板

### 模板 A：Bug 修复

```
【环境】Python 3.11, FastAPI 0.110
【文件】src/api/users.py（已附上完整代码）
【问题】调用 /users/{id} 接口时，当 id 不存在时返回 500 而非 404
【期望】返回标准 404 响应，body 为 {"detail": "User not found"}
【限制】
- 只修改 get_user 这一个函数
- 不要修改任何其他函数
- 不要引入新的依赖
【输出格式】只输出修改后的 get_user 函数，加上修改说明（1-2句话）
```

### 模板 B：新功能开发

```
【背景】现有代码库结构如下：[简述或附代码]
【需求】实现 [具体功能]，要求：
  1. [具体要求1]
  2. [具体要求2]
【约束】
- 复用现有的 [工具函数/类]，不要重复实现
- 不要修改现有接口签名
- 函数要有类型注解
【先做】先列出实现方案（不写代码），我确认后再实现
```

### 模板 C：代码审查

```
请审查以下代码，重点关注：
1. 潜在的运行时错误
2. 边界条件遗漏
3. 性能明显低效的地方

不需要：
- 风格建议
- 重构建议
- "可以考虑用XXX更优雅"这类评论

对每个问题给出：[行号] + [问题描述] + [修复建议]
如果代码没有问题，直接说"未发现问题"。

[粘贴代码]
```

## 十、原则速查卡

```
✅ 永远提供版本号
✅ 粘贴实际代码，而非描述代码
✅ 一次只做一件事
✅ 明确"不要做什么"
✅ 让 AI 先说方案，再写代码
✅ 要求 AI 在不确定时说出来
✅ 大文件用 diff 模式输出
✅ 生产代码禁止"顺手优化"

❌ 不要用模糊的业务语言描述技术需求
❌ 不要让 AI 自行决定技术选型
❌ 不要一次性要求完整系统
❌ 不要忽略 AI 的"我认为"/"可能"等不确定信号
❌ 不要依赖 AI 了解你的项目全局
```
